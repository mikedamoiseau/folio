- **File**: [src-tauri/src/commands.rs](/Users/mike/Documents/www/folio/src-tauri/src/commands.rs:3994), [src-tauri/src/db.rs](/Users/mike/Documents/www/folio/src-tauri/src/db.rs:430), [src-tauri/src/commands.rs](/Users/mike/Documents/www/folio/src-tauri/src/commands.rs:716)

- **Severity**: BLOCKING
- **Fixable**: YES

2. **`bulk_add_tag` writes to columns that do not exist in `book_tags`**
- **File**: [src-tauri/src/db.rs](/Users/mike/Documents/www/folio/src-tauri/src/db.rs:461), [src-tauri/src/db.rs](/Users/mike/Documents/www/folio/src-tauri/src/db.rs:144)
- **What** can go wrong: the first call to the new bulk-tag command will fail at runtime with a SQLite error, because it inserts `(id, book_id, tag)` into `book_tags`, but that table only has `(book_id, tag_id)`.

- **Impact** if it happens: the exposed Tauri command is broken on arrival; any future UI or automation that tries bulk tagging will fail completely.
- **Fix** recommendation: mirror the existing single-tag flow: resolve/create the tag in `tags`, get its `id`, then insert `(book_id, tag_id)` into `book_tags` with `INSERT OR IGNORE`.
- **Severity**: BLOCKING
- **Fixable**: YES

## [Claude ‚Äî Implementer] Round 1

NEEDS_FIX: Two confirmed blocking bugs in bulk operations ‚Äî orphaned files on delete and wrong schema on tag insert

Both issues raised by Codex are real and verified against the source:

**Issue 1: Bulk delete skips file/cache cleanup ‚Äî AGREE (BLOCKING)**

The existing `remove_book` (commands.rs:716-763) performs four cleanup steps after the DB delete:
1. Evicts the EPUB archive cache entry (line 735-738)
2. Deletes the imported physical file (lines 742-755)
3. Removes the image cache directory (lines 757-761)
4. Logs activity with per-book title (lines 723-730)

The new `bulk_delete_books` command (line 3988) bypasses all of this and only calls `db::bulk_delete_books`, which is a bare `DELETE FROM books`. This is a storage leak ‚Äî imported files, extracted images, and cached EPUB data all remain on disk.

**Fix:** Before the DB delete, fetch all targeted books via `db::get_book()` for each ID. After the transactional DB delete, loop through the fetched books and perform the same cleanup as `remove_book`: evict epub cache, delete imported files, remove image cache dirs. This can stay in a single transaction for the DB portion.

**Issue 2: `bulk_add_tag` uses wrong schema ‚Äî AGREE (BLOCKING)**

The `book_tags` table (db.rs:144-148) has columns `(book_id, tag_id)` referencing the `tags` table. But `bulk_add_tag` (db.rs:461-468) inserts `(id, book_id, tag)` ‚Äî three columns that don't exist in that schema. This will produce a SQLite error on every call.

The existing single-tag flow uses `add_tag_to_book` (db.rs:1239) which correctly inserts `(book_id, tag_id)`, and the tag itself is resolved/created in the `tags` table first.

**Fix:** Mirror the existing flow: resolve the tag name to its `tags.id` (INSERT OR IGNORE into `tags` first), then INSERT OR IGNORE `(book_id, tag_id)` pairs into `book_tags`.
